## 6. Tự đánh giá & rút kinh nghiệm (tùy chọn)

> Phần optional, không tính điểm — nhưng giúp thầy hiểu hơn về quá trình tôi làm việc.

### Điều bạn làm tốt:

> Tôi chủ động đi trước bước **dựng nền tảng** cho cả team ở Phase 1 (ticket #16 — Project Setup): cấu trúc 3-layer Clean Architecture, DI per-layer (`getIt`), env config qua `--dart-define`, routing tập trung, mock repository layer để các thành viên khác plug vào ngay. Nhờ vậy các ticket UI Phase 1 của nhóm không phải tự dựng kiến trúc — chỉ cần follow scaffold order trong `.github/instructions/`. Ở Phase 2, khi backend không cung cấp đúng spec đã hứa (password flow), tôi chủ động sync lại để chốt scope mới (OTP + PKCE) thay vì block ticket — vẫn deliver được 2 luồng auth production-ready trong cùng ticket #50.

### Khó khăn bạn gặp phải:

> Khó khăn lớn nhất là **làm việc với API spec lệch thực tế và 2 host backend khác nhau**. Ticket #50 ban đầu lên kế hoạch quanh password management nhưng BE không có; phải reset hoàn toàn approach sang OTP + PKCE giữa sprint. Hai host `auth-api` và `helpdesk-api` đòi headers khác nhau (`X-Stack-Project-Id`, `X-Stack-Access-Type`, `X-Tenant-Id`) — Dio interceptor phải phân biệt host để gắn header đúng, nếu nhầm là 401/403 ngay. Phía Google PKCE thì phải mở keystore release riêng để debug-build có SHA-1 hợp lệ với OAuth client. Ngoài ra, ở ticket #58 nội dung Knowledge Source có nhiều biến thể (file / web URL / whole-site SSE crawl / database query), mỗi loại endpoint shape khác nhau — tôi và @xiaohonsu phải align nhiều lần để chia scope sạch (tôi nhận DB query + SSE crawl + delete; @xiaohonsu nhận web import + file upload core).

### Bạn sẽ làm khác đi điều gì nếu được làm lại:

> (1) **Lock spec backend bằng contract** trước khi bắt tay Phase 2 — yêu cầu BE commit OpenAPI / Apidog cho **toàn bộ** endpoint của ticket trước khi mobile estimate, để không bị mid-sprint pivot như vụ password→OTP. (2) **Tách Dio host config ra một module riêng có integration test** ngay từ đầu (`auth_dio` vs `helpdesk_dio`), thay vì để lẫn trong interceptor — sẽ ít bug 401 do nhầm header. (3) Với ticket co-assigned như #58, kick-off bằng **whiteboard scope split** ngay ngày đầu (data model nào của ai, store nào ai sở hữu, route nào ai mount), thay vì để work tự nhiên đụng nhau giữa chừng và phải rebase nhiều vòng.
